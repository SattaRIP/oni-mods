THE CRASH

Two related IndexOutOfRangeException stack traces in Player.log:

 1. Constructable.OnSpawn_Patch1 (path 1) — fires when a wider
    bridge enters its CONSTRUCTION phase. Stack ends in:
      Diggable.IsDiggable(cell)
      Constructable.<PlaceDiggables>b__0(offset_cell)
      BuildingDef.RunOnArea(cell, orientation, callback)
      Building.RunOnArea(callback)
      Constructable.PlaceDiggables()
      Constructable.OnSpawn_Patch1

 2. Diggable.OnCleanUp_Patch1 (path 2) — same exception, but fires
    when ANY nearby dig job is destroyed. Loops on every dig event
    once an under-construction wider bridge is in the world.

The specific instance in your log: HPARailBridge2UnderConstruction
at world position (0.50, 0.00, -10.50). That x≈0.5 is the right
edge of column 0 — i.e. the building was placed with its anchor
at the LEFT EDGE of the asteroid.

ROOT CAUSE

Wider bridges (4 cells wide, 5 cells wide) use centered cell
offsets like [-1, 0, 1, 2] for 4-wide or [-2, -1, 0, 1, 2] for
5-wide. When the player places one near the LEFT edge of the
asteroid, the leftmost cell of the footprint lands at a column
that doesn't exist in the world (column -1 or -2), and ONI's
Grid array access throws IndexOutOfRangeException.

This is a real bug in our config — vanilla 4-wide buildings
(turbines, doors, etc.) either:
 - use non-negative offsets so they extend only RIGHT of the
   anchor, OR
 - have placement validation that prevents anchor positions
   where the footprint would go off-grid.

We have neither.

NOT RELATED to: the rotation work, the BridgeRotationHandler
component, the Subscribe-based event handling, the cross-
contamination cleanup. Those are orthogonal — this crash would
hit even with the rotation code stripped out entirely.

THREE FIX OPTIONS (pick one tomorrow)

OPTION A — Change cell offsets to non-negative (cleanest, no
Harmony):
  In each Config's CreateBuildingDef, after CreateBuildingDef
  call, override def.PlacementOffsets to be {(0,0), (1,0),
  (2,0), (3,0)} for 4-wide. Side effect: building extends RIGHT
  of the placement cursor instead of being centered on it.
  Visual offset (BridgeHelpers.StretchKanim's offsetX) will
  need a corresponding shift.

OPTION B — Harmony prefix on Constructable.PlaceDiggables that
catches the exception silently. Trivial in code but reintroduces
the Harmony-dynamic-method crash family we just escaped. NOT
RECOMMENDED.

OPTION C — Harmony prefix on Diggable.IsDiggable that guards
with Grid.IsValidCell and returns false for off-grid cells.
Same Harmony risk as B but smaller surface area.

I'd go A first. If we can't easily change anchor without
breaking the rotation/stretch math, C as fallback.
